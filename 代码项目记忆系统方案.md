# 代码项目记忆系统方案

将个人知识库的记忆系统体系迁移到代码项目管理中，配合工作流层作为人机交互界面，构成完整的 vibe coding 工作范式。

---

## 核心洞察

现有的代码索引/知识图谱方案（Code Map、RepoAgent、CodeGraph、DeepWiki、Karpathy LLM Wiki 等）都在做"给代码建索引"，本方案做的是"给代码项目建记忆系统"。索引回答"东西在哪"，记忆回答"是什么、为什么、和什么有关、怎么演化来的"。

关键的认知跃迁：代码本身不是记忆的对象，注释才是。代码是原始感觉输入，注释是对代码的语义编码。大模型不需要"记住"代码怎么写，只需要记住代码"是什么、为什么、和什么有关"。这从根本上解决了大模型上下文窗口不够的问题——不是压缩代码，而是换了记忆对象。


## 记忆系统架构

### 原始层：代码

不参与记忆，只在需要时按需调取。地位类似人脑的感觉输入——大量、未加工、时刻在变。

### 浅层记忆：三层注释体系

注释必须是多层次的，不同代码层级的注释关注不同的东西：

1. 代码块注释 — 关注实现原理（Mechanism）。粒度最细，数量最大，单独看意义有限。
2. 模块注释 — 关注模块目标和技术选型（Decision/Architecture）。是做决策时需要调取的层次。
3. 系统级注释 — 关注整体架构与模块间的连通方式（Architecture/Pipeline）。是理解项目全貌的层次。

三层注释形成压缩链路：大模型日常只需读系统级和模块级注释，定位到具体问题时才下探到代码块注释，再需要时才去读原始代码。

**浅层记忆的基本元素和 X 轴标签**

好的、针对具体系统的代码规范和专用术语（目的、方式、方法名、变量名、依赖等），构成浅层记忆的基本元素和 X 轴标签。这些都是软件工程的老生常谈，但在这个体系里它们被赋予了新的角色——不再只是"编码规范"，而是"记忆编码规则"。

**注释作为一等公民**

注释一旦成为记忆对象，就必须被当作一等公民来维护。不再是"写给人看的备注"，而是"项目认知系统的数据源"。注释质量直接决定记忆体质量。这反过来倒逼代码规范和注释规范的严格执行，形成正循环。

### 深层记忆：涌现侧 + 牵引侧

**涌现侧（自底向上）**

对注释做 embedding 向量化，通过语义相似度发现注释之间的隐含关联。发现哪些模块虽然在代码组织上离得远，但在语义上谈的是同一件事。数据自己说话，无需人工指定。

**牵引侧（自顶向下）**

用两类质心目标反向索引注释：

- 业务目标：系统要实现什么业务价值
- 系统目标：稳定性、兜底、性能、安全等非功能性要求

每条注释被度量与这些目标之间的距离，形成全景图——"这个模块对哪个目标贡献最大、哪个目标目前覆盖最薄弱"。

**两侧叠加**

涌现侧 + 牵引侧叠加，形成项目的认知地图。大模型读这张地图，等于瞬间获得"在这个项目里工作了半年的工程师"的理解水平。


## 记忆系统的三大使用场景

### 场景一：检验代码与项目的差距（验证闭环）

基于代码构造的记忆（自底向上）与基于项目建造记录形成的项目描述（自顶向下），从不同途径对系统进行描述。两者进行校验与互锁：

- 能对上的 → 实现到位
- 代码记忆有但项目描述没有 → 冗余代码
- 项目描述有但代码记忆没有 → 未实现的功能

给 vibe coding 加了一层最强锁——解决"跑起来了但不知道跑的对不对"的核心问题。

### 场景二：节省上下文（效率优化）

每次对话起始载入项目描述和记忆体，快速理解项目目标和实际情况，不用每次重新读代码库。把大模型的推理资源用在理解和思考上，而不是阅读细节上。

### 场景三：系统迭代修改评估（安全性保障）

- 通过记忆索引快速识别相关可能的影响点
- 通过注释反索引具体代码块进行阅读，理解接口数据格式和对接实现方式，避免"想当然"的改动
- 通过语义索引"联想"有可能受影响的模块，避免漏改与跨模块/系统盲区
- 效果：一次改对 & 一次改全，节省调试时间，避免拆东墙补西墙

语义索引的联想能力覆盖了调用关系图覆盖不了的盲区——代码的显式依赖图只告诉你调用关系，但语义相似度能告诉你"这两个模块虽然没有直接调用，但它们在处理同一个数据格式的不同阶段"。


## 完整的 Vibe Coding 工作范式

三个使用场景合在一起，给 vibe coding 补上了软件工程的质量闭环：

- 验证（对不对）— 场景一
- 效率（快不快）— 场景二
- 安全性（改全没有）— 场景三

这个闭环不依赖人的经验——经验被外化到记忆体里。

配合工作流层作为人机交互界面（用户通过自然语言描述意图 → 系统渐进式形式化 → 生成可执行的工作流），构成完整的 vibe coding 工作范式：

- 记忆系统解决"AI 理解项目"的问题
- 工作流层解决"人表达意图"的问题
- 两者之间通过结构化的注释/标签体系桥接


## 与现有方案的比较

| 方案 | 做到了什么 | 没做到什么 |
|------|-----------|-----------|
| Code Map / Repomix | 静态结构提取，压缩到5-10% | 没有语义理解，没有关联发现 |
| RepoAgent | 自动生成文档 | 本质是"给代码写说明书"，没有记忆层级 |
| CodeGraph / FalkorDB | 知识图谱，显式依赖关系 | 只有调用关系，没有语义关联 |
| Karpathy LLM Wiki | 增量编译结构化wiki+双链 | 停留在浅层记忆，没有质心牵引 |
| DeepWiki | 产品化最好的自动知识库 | 被动索引，没有验证闭环 |
| CLAUDE.md / AGENTS.md | 手写结构化上下文 | 纯手工，不可扩展 |
| **本方案** | 三层注释→双层涌现→三大使用场景 | 完整的记忆系统+验证闭环+工作流交互 |


## 理论基础

本方案是"记忆系统的分层涌现模型"在代码工程领域的实例化：

- 代码 = 感觉输入
- 三层注释 = 感觉记忆→工作记忆→长期记忆的压缩编码
- 涌现侧（向量化+相似度）= 工作记忆中的自动关联
- 牵引侧（目标反向索引）= 长期记忆中核心信念对信息的主动吸附
- CLAUDE.md = 本方案的极简手工原型
