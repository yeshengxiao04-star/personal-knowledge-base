## 一、系统总体架构

**原作者原话：** "RAG不是软件，是一套系统，你要自己用python开发。我的方法是xyz三维，加一颗神经束。X是按事件分类，Y是结构，Z是拓扑。"

系统是一套Python自建的个人知识管理RAG，核心设计特征：
- 每条笔记同时在三个正交维度（X/Y/Z）上被标注定位
- 笔记内部遵循统一的认知加工模板
- episodes（知识片段）经Thread Dossier Builder自动编织为连贯知识线索（"神经束"）
- 检索链路：三维标签预筛选 → 向量召回 → Rerank重排 → LLM生成

### 复现方案的架构分层

原作者全部用Python自建。复现方案引入 **Obsidian** 作为数据管理层，形成双层架构：

```
┌─────────────────────────────────────────────┐
│  Obsidian层（零代码，日常使用）                  │
│  · 笔记撰写与模板填充（Templater插件）           │
│  · 三轴标签管理（YAML frontmatter + 层级tags）  │
│  · 标签统计与切面查询（Dataview插件）            │
│  · 日常知识图谱浏览（原生Graph View）            │
├─────────────────────────────────────────────┤
│  Python层（检索与生成）                         │
│  · 读取vault目录，解析.md + YAML frontmatter   │
│  · Embedding生成 + 向量存储                    │
│  · 标签预筛选 + 向量召回 + Rerank + LLM生成     │
│  · Thread Dossier Builder（后期）              │
│  · 精确语义拓扑可视化（如需，后期）               │
└─────────────────────────────────────────────┘
         ↕ 共享同一个vault文件目录，无需同步机制
```

两层通过文件系统直接共享数据——Obsidian的vault就是一个普通文件夹，Python脚本直接读取同目录下的.md文件。

## 三、笔记内部模板

**【确认】** 截图5-6展示了一条完整笔记（REM睡眠与药物干预）的内部结构：

### 笔记属性（元数据头）

**【确认】** 截图5中的原始格式：
```
title:    REM 睡眠的生理功能与药物干预的双刃剑效应
date:     2026-01-29 17:48
tags:     v/Mechanism × Health/Medicine × Health/Psy × Health/Biohack × z/Node ×
```

**Obsidian兼容的YAML frontmatter写法：**
```yaml
---
title: REM 睡眠的生理功能与药物干预的双刃剑效应
date: 2026-01-29 17:48
tags:
  - y/Mechanism
  - x/Health/Medicine
  - x/Health/Psy
  - x/Health/Biohack
  - z/Node
---
```
Obsidian原生支持层级标签（`x/Health/Medicine`在标签面板中自动展开为树状结构），Dataview可直接查询，Python端用python-frontmatter库解析。三端一致，无需格式转换。

### 笔记正文模板（5段式）

**1. 🎯 场景与痛点 (Context & Anchor)**
- 触发场景：记录这条知识的起因和上下文
- 关键阻碍：当前认知的盲区或误区
- 原始诉求：想要解决的核心问题

**2. 🔬 核心知识与底层原理 (First Principles)**
- 核心特征/机制的系统描述
- 功能设定（列举关键功能点）
- 因果链条（如：某药物如何破坏某机制）

**3. ⚡ 详细执行方案 (Execution & Code)**
- 质量评估指标（如何判断执行效果）
- 优化方向（具体可行的改进路径）

**4. 🚧 避坑与迭代路径 (Debugging & Iteration)**
- 认知修正（需要纠正的常见误区）
- 依赖陷阱（执行中容易踩的坑）

**5. 💡 启发与思维模型**
- 篇幅总结（对这条知识的元认知反思）
- 代偿机制（系统层面的洞察和跨域映射）

**【推测】** 这个模板不是所有笔记都严格遵循的硬性模板，可能根据笔记类型（Y轴标签不同）有变体。但核心框架一致：从场景出发 → 提炼原理 → 落地执行 → 记录踩坑 → 抽象反思。

## 四、Thread Dossier Builder（"神经束"）

**【确认】** 截图1展示了Thread Dossier Builder的运行状态：

```
1. episodes_built: 11
   从之前的 23 掉到 11，说明碎片被压下来了

2. orphan_context_blocks: 12
   说明很多纯解释块没有再硬生生 episode，这很对

3. Ready For Thread Dossier: true = 11
   说明系统已经能把事件簇捋到 thread 案卷层
```

**结论先给：可以进入 Thread Dossier Builder 了。**

**但有一个前提：Thread Dossier Builder 不能把 episode 当"最终真理"，而要允许 episode 在 dossier 内继续合并/重组。**

### 推测的工作机制

**【推测】** 基于截图信息反推的处理流程：

1. **原始笔记 → Episode生成：** 系统将相关笔记聚合为episodes（从23条聚合为11条，压缩率约52%）
2. **孤立块识别：** 识别出12个orphan_context_blocks——不属于任何episode的纯解释性内容块
3. **Thread Dossier编织：** 将episodes编织为连贯的知识线索（thread），形成"案卷"（dossier）
4. **持续重组约束：** dossier不是终态，episodes在dossier内可以继续合并和重组

**【推测】** 这一层可能是定期批处理运行的（非实时），类似离线整理过程。

## 六、检索链路

**【确认】** 截图7展示了检索结果页面，包含性能指标：

```
Rerank 4.4s + LLM 13.3s = 17.7s
```

### 推测的检索管线

**【推测】** 完整检索流程大致为：

```
用户Query
  ↓
1. 三维标签预筛选（基于X/Y/Z标签缩小候选集）
  ↓
2. 向量相似度召回（在候选集内做embedding匹配）
  ↓
3. Rerank重排（4.4s）——可能使用cross-encoder或专用reranker模型
  ↓
4. LLM生成（13.3s）——基于重排后的top-k结果生成最终回答
  ↓
输出
```

**【确认】** 截图7中LLM生成的内容涉及EMDR疗法与REM睡眠的关系，内容质量较高，能做跨笔记的知识综合。说明检索端能正确召回多条相关笔记并让LLM做综合推理。

**【推测】** 检索生成的LLM可能是Claude Opus（截图7顶部有疑似"Claude Opus 1.0"的字样），reranker模型不确定。

## 九、工程复现时的关键决策点

以下是复现时需要自行做出的工程决策：

### 9.1 必须决定的

1. **标签体系是硬编码还是可扩展？** Y轴7个类目、Z轴4个类目是否固定，还是允许随使用演进。Obsidian的Tag Wrangler插件支持批量重命名和合并标签，为后期演进留了空间。
2. **每条笔记是单标签还是多标签？** 截图5的笔记挂了多个标签（v/Mechanism + Health/Medicine + Health/Psy + Health/Biohack + z/Node），说明至少X轴是多标签的。Y轴和Z轴是否也允许多标签？
3. **笔记存储格式：** 确定使用Obsidian vault，即Markdown with YAML frontmatter。标签写在frontmatter的tags字段中，Obsidian和Python两侧均可原生解析。
4. **Thread Dossier的触发和执行频率：** 是每次新增笔记后增量运行，还是定期批处理？
5. **知识图谱可视化的精度需求：** Obsidian原生Graph View基于文件间的双向链接（`[[]]`）构建拓扑，而群友截图2-4的图谱可能基于语义相似度或标签共现关系。两者的拓扑结构不同。需要决定：
   - **如果日常浏览够用：** 直接用Obsidian原生Graph View，通过在笔记间手动或脚本添加`[[]]`双向链接来构建拓扑，按标签染色区分主题簇。优点是零代码、实时更新、交互体验好。
   - **如果需要精确的语义拓扑：** 仍需Python端用向量相似度或共现矩阵计算节点间关系，用NetworkX + pyvis生成独立的图谱可视化。优点是拓扑关系由数据驱动而非人工链接。
   - **折中方案：** 日常用Obsidian Graph View浏览，定期用Python脚本扫描vault并自动在笔记间插入`[[]]`链接（基于embedding相似度超过阈值），让Obsidian图谱的拓扑由数据驱动更新。这样两层都有图谱能力且保持一致。

### 9.2 可以先简化的

1. **Reranker：** 初期可用简单的余弦相似度排序，后期再引入cross-encoder。
2. **Thread Dossier：** 这是整个系统中最复杂的部分，初期可以跳过，先把三轴标签 + 模板化笔记 + 基础检索跑通。
3. **精确语义拓扑：** 初期用Obsidian原生Graph View足够，不需要单独开发可视化。

### 9.3 复现的建议优先级

```
Phase 1: Obsidian vault搭建（零代码）
         → 安装Templater + Dataview插件
         → 定义三轴标签分类表（X/Y/Z各轴的完整类目）
         → 创建Templater笔记模板（五段式框架 + 交互式标签选择）
         → 用Dataview写标签统计查询（复现截图8的效果）
         → 手工写5-10条测试笔记验证完整流程
         
Phase 2: Python读取vault + Embedding（开始写代码）
         → 扫描vault目录，用python-frontmatter解析.md文件
         → 提取YAML标签（X/Y/Z三轴）+ 笔记正文
         → 生成embedding，存入向量数据库
         → 实现基于标签的过滤检索（标签预筛选 + 向量召回）
         
Phase 3: Rerank + LLM生成
         → 补全完整的检索-生成管线
         → 调通端到端流程：Query → 标签筛选 → 向量召回 → Rerank → LLM输出
         
Phase 4: 图谱增强（可选）
         → 基于Phase 2的embedding数据，计算笔记间语义相似度
         → 自动在vault笔记中插入[[]]双向链接
         → 或用NetworkX + pyvis生成独立的语义拓扑图
         
Phase 5: Thread Dossier Builder
         → 实现episode压缩和自动编织
         → 定义触发条件和执行频率
```

## 十、与"记忆涌现模型"的对照备注

供工程复现过程中参考理论模型的对应关系：

| 群友系统组件 | 记忆涌现模型对应概念 | 差异 |
|-------------|-------------------|------|
| X轴主题标签 | 无直接对应（模型不做人工分类） | 群友显式分类 vs 模型主张涌现 |
| Y轴结构标签 | 4.2节 价值取向显式化 | 群友人工固定 vs 模型主张自演化 |
| Z轴拓扑标签 | 3.7节 向量-图混合架构中的图结构 | 群友人工标注拓扑角色 vs 模型自动生成连接图谱 |
| Thread Dossier | 路径三：重组（深层→浅层写入） | 群友批处理编织 vs 模型的Semantic Sweep自动重组 |
| 笔记模板 | 无直接对应 | 模型Layer 0是无结构的原始记忆 |
| Rerank + LLM | 3.3节 检索路径（自上而下锚点拉取） | 群友传统rerank vs 模型从深层质心开始匹配 |
| 知识图谱可视化 | 过饱和溶液结晶模型的空间投影 | 群友显式网络图 vs 模型向量空间中的密度结构 |

**核心启发：** 群友的Y轴标签体系可作为涌现模型"价值取向显式化"的冷启动实现。先用人工标注建立ground truth，后续验证HDBSCAN涌现聚类是否能自动复现Y轴分类。

**Obsidian作为实验平台的额外价值：** Phase 2跑通embedding后，同一批笔记数据同时具有人工标签（Obsidian侧）和向量表征（Python侧）。可以直接做对照实验：HDBSCAN在向量空间上的自动聚类结果 vs Y轴人工标签的ARI对齐度，即验证H12（向量聚集能作为涌现载体）。Obsidian的Dataview查询能力使得ground truth的统计和抽样非常方便。
